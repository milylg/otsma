# 什么是类加载

通俗地说，将一个类的字节码文件加载到内存中的运行时数据区，产生一个Class对象，用于提供访问这个类的数据接口。

# 类加载有哪些机制

  • 缓存机制
  • 双亲委派机制

# 缓存机制

在类加载的过程中，执行搜索字节码文件之前，首先判断内存中是否已经加载了该字节码文件，这样的操作会减少重复加载，避免了时间消耗。

# 双亲委派机制

每个类型的类加载器都有自己的加载范围。双亲委派机制的大致流程是这样的：首先是由自定义的类加载器负责加载，但是它不是优先查找自己的加载范围的字节码文件，而是先委托它的父加载器去加载，直到父加载器没有向上委托的加载器时，才开始尝试在父加载器自己所扫描的文件夹范围中寻找该字节码文件。如果存在该字节码文件，则加载该文件到内存中；如果在父加载器所负责的文件范围内不存在该字节码文件，则让子加载器尝试去加载，子加载器会在自己所属范围内搜索指定要加载的文件；如果没有，向下委托，直到自定义的加载器。当所有的加载器都没有找到该字节码文件时，则加载失败。双亲委派机制之所以这样设计，一方面是为了保护API不被篡改，另一方面避免重复加载字节码文件。

# 自定义类加载器

具体实现如下：

  • 初始化类加载的目录
  • 重写ClassLoader的findClass(String)方法
代码清单 - 自定义类加载器 
public class CustomClassLoader extends ClassLoader {

    /**
     * 字节码文件绝对路径
     */
    private String classPath;

    public CustomClassLoader(String classPath) {
        this.classPath = classPath;
    }

    private byte[] loadByte(String name) throws IOException {
        name = name.replaceAll("\\.","/");
        FileInputStream fis = new FileInputStream(classPath + "/" + name + ".class");
        int len = fis.available();
        byte[] data = new byte[len];
        fis.read(data);
        fis.close();
        return data;
    }

    @Override
    protected Class<?> findClass(String name) {
        try {
            byte[] data = loadByte(name);
            return defineClass(name, data, 0, data.length);
        } catch (IOException e) {
            System.out.println("exception : " + e.getMessage());
        }
        return null;
    }
}

# 双亲委派机制实现

# loadClass(String)

Loads the class with the specified binary name.This method searches for classes in the same manner as the {@link#loadClass(String, boolean)} method. It is invoked by the Java virtual machine to resolve class references. Invoking this method is equivalent to invoking {@link #loadClass(String, boolean) loadClass(name,false)}.

方法的功能是加载指定的二进制文件，它会执行默认不解析字节码对象的重载方法，由虚拟机中的方法去执行解析操作。

# loadClass(String, boolean)

Loads the class with the specified binary name. The default implementation of this method searches for classes in the following order:

  • Invoke findLoadedClass(String) to check if the class has already been loaded.
  • Invoke the loadClass method on the parent class loader. If the parent is null the class loader built-in to the virtual machine is used, instead.
  • Invoke the findClass(String) method to find the class.

If the class was found using the above steps, and the resolve flag is true, this method will then invoke the resolveClass(Class) method on the resulting Class object. Subclasses of ClassLoader are encouraged to override findClass(String), rather than this method. Unless overridden, this method synchronizes on the result of getClassLoadingLock method during the entire class loading process.

这个方法的功能仍然是加载指定的二进制文件，它比上述的方法多一个参数，用于配置是否解析字节码对象。该方法的默认实现按顺序对下列条件执行相应的搜索方法。

  • 首先检查字节码文件是否已经被加载，使用findLoadedClass方法。如果没有被加载，进入下列条件。
  • 如果当前加载器有父加载器，那么就执行父类加载器的loadClass方法，向上委托。如果父类为空，则使用虚拟机内置的类加载器Bootstrap Class Loader的findBootstrapClassOrNull方法。
  • 如果父类加载器找不到指定的字节码文件，那么就执行当前类加载器的findClass方法搜索字节码文件

如果字节码文件在最后一个条件中执行搜索时被找到，且解析字节码对象的标志为true，那么执行resolveClass(Class)方法对字节码对象进行解析操作（底层native方法）并返回字节码对象。

建议

推荐子类加载器重写了父加载器的findClass(String) 方法，而不是使用loadClass(String,boolean)方法。除非loadClass(String,boolean)被重写，否则它在整个类加载过程中需要同步getClassLoadingLock方法返回的结果。通俗地说，如果要重新自定义加载字节码的策略，而不是使用双亲委派机制来加载字节码，则需要在子类加载器中重新定义loadClass(String,boolean)方法，即重写loadClass方法。

加载字节码文件的源码实现 
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    
    synchronized (getClassLoadingLock(name)) {
        // First, check if the class has already been loaded
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            long t0 = System.nanoTime();
            try {
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // ClassNotFoundException thrown if class not found
                // from the non-null parent class loader
            }

            if (c == null) {
                // If still not found, then invoke findClass in order
                // to find the class.
                long t1 = System.nanoTime();
                c = findClass(name);

                // this is the defining class loader; record the stats
                sun.misc.PerfCounter.getParentDelegationTime().addTime(t1 - t0);
                sun.misc.PerfCounter.getFindClassTime().addElapsedTimeFrom(t1);
                sun.misc.PerfCounter.getFindClasses().increment();
            }
        }
        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}

# 类加载器的类型

从上到下按父与子的顺序排列：

  • Bootstrap ClassLoader 启动类加载器，主要负责加载Java核心类库，%JRE_HOME%\lib下的rt.jar、resources.jar等字节码文件。
  • Extention ClassLoader 标准扩展类加载器，主要负责加载目录%JRE_HOME%\lib\ext目录下的jar包和字节码文件。
  • Application ClassLoader 应用类加载器，主要负责加载当前项目的classpath下的所有字节码文件。
  • User ClassLoader 用户自定义类加载器，加载指定路径的字节码文件。

注意

这里的父加载器和子加载器在实现层面不是继承关系，而是组合关系，子类复用了父类的一些方法，符合组合复用的设计原则。

# 类加载器的应用

  • 加密，字节码文件可以通过反编译获得java源文件信息。为了保护代码的信息不外泄，将字节码文件进行加密，在类加载的过程中，首先对字节码文件进行解密操作，接着进行类加载操作。
  • 加载其他来源的字节码文件，来源可以是网络，本地目录，数据库。
