不使用操作符实现加法

Gen 2020-10-17 剑指offer

# 题目描述

题目：写一个函数，求两个整数之和，要求在函数体内不得使用＋、－、×、÷四则运算符号。

# 测试用例

  • 输入都是正整数
  • 输入正整数和负整数，负整数绝对值小于正整数
  • 输入正整数和负整数，负整数绝对值大于正整数
  • 输入一对相反数

# 我的解决方案（不完全正确）

代码仅仅只能处理正数相加的问题域，不能处理负数，因此，并不能使所有测试通过。

public static int addByBit(int a, int b) {
    int sum = 0;
    int carry = 0;
    int indexOfBit = 0;
    int bitOfA, bitOfB;
    while ((a >> indexOfBit) != 0 || (b >> indexOfBit) != 0 || carry == 1)
    {
        bitOfA = a >> indexOfBit & 1;
        bitOfB = b >> indexOfBit & 1;
        sum |= (bitOfA ^ bitOfB ^ carry) << indexOfBit;
        carry = carry == 1 ? bitOfA | bitOfB : bitOfA & bitOfB;
        indexOfBit ++;
    }
    return sum;
}

# 正确的解答思路

Key words:

  • 进位
  • 非进位和：当前位相加，只取一位低位，进位舍弃，也就是暂不处理进位值。

For Example:

1750 + 2365 = ?

非进位和为：3015

​ 1 7 5 0

​ 2 3 6 5

​ 3 0 1 5

进位为：1100

​ 1 7 5 0

​ 2 3 6 5

​ 1 1 0 0

此时，1750 + 2365转换为了3015 + 1100，进行下一个上述逻辑运算

结束条件为进位为0时，整个运算过程结束。

现在将十进制的运算逻辑迁移到二进制的运算逻辑中。

For Example：

15 + 17 = ? 两个数字使用 numberOne， numberTwo 进行标识

非进位和为： 1 1 1 1 0 使用sum 标识

​ 0 1 1 1 1

​ 1 0 0 0 1

​ 1 1 1 1 0

可以用运算符表示为 1 1 1 1 ^ 1 0 0 0 1 即 sum = numberOne ^ numberTwo

进位为：1 0 使用carry 标识

​ 0 1 1 1 1

​ 1 0 0 0 1

​ 0 0 0 1 0

可以用运算符表示为 (1 1 1 1 & 1 0 0 0 1) << 1 即 carry = (numberOne & numberTwo) << 1

此时进位carry并没有置为零（继续逻辑运算的条件），所以接下来，需要进行下一轮逻辑运算，

(1 1 1 1 + 1 0 0 0 1) 转变成了(1 1 1 1 0 + 1 0)的加法运算

在进行下一轮逻辑运算之前需要更新 numberOne 和 numberTwo 的值

即

​ numberOne = sum

​ numberTwo = carry

现在，继续下一次逻辑运算，直至carry = 0 时，获取结果。

将代码翻译上述逻辑处理流程，如下：

public static int add(int a, int b) {
    int sum, carry;
    do {
        sum = a ^ b;
        carry = (a & b) << 1;
        a = sum;
        b = carry;
    } while(b != 0);
    return a;
}

# 思想的升华

为什么我自己的逻辑处理不能够处理负数，而上述的十进制运算逻辑迁移却可以应用到二进制中？

首先，我的处理方式也是从十进制的加法逻辑中迁移出来并应用到二进制的加法逻辑运算中，在这一点上我的思考方式与正确的思考方式是一致的。

现在我能够看出是我的思考的角度与有效的处理方式是不同的，我按照1bit位1bit位的计算，而有效的解答所处理的是将一个数字作为一个整体来处理的。我知道这是我们的思考维度的不同，而我不知道的是为什么这种方式可以处理负数，造成这种差异的原因是什么？

我使用bit位进行运算，忽略了正数和负数相加以及非正数之间的相加，忽略了这些问题，会导致运算出现死循环的问题（在我的解决方法的代码上）。在计算机底层，CPU只会加法，所有的操作符运算都可以使用加法来转换。计算机科学家为了处理上述的问题，又提出了两个概念------反码和补码。这些概念早已封装在计算机系统的底层中。

概念提要

  • 反码解决了正负数相加的问题

  • 补码解决了+0 和 -0的问题

原码当一个数占8bit的空间，低7位标识为数值，最高位为符号位，0为正数，1为负数。表达范围为**[-127, 127]**

反码基于原码计算得来的，正数的反码是它本身。但负数的反码是其原码的非符号位取反后的结果

补码是基于原码和反码派生出来的，正数的补码是它本身，负数的补码是它的反码加上1的结果，表达范围**[-128,127]**

For Example : -14 原码(1000 1100) 反码(1111 0011) 补码(1111 0100)

运算方式

  • 反码

    • 符号位与数值一起参加运算

    • 反码的符号位相加后，如果有进位出现，则要把它与结果的最低位相加

    • 用反码进行运算，结果也是反码，对于结果是正数来说，反码就是它本身，如果是负数，则要将非符号位的位进行取反，才是其结果的原码。

  • 补码

    • 符号位与数值一起参加运算
    • 补码的符号位相加后，如果有进位出现，则要把进位舍弃
    • 用补码进行运算，结果也是补码，对于结果是正数来说，补码就是它本身，如果是负数，则要将该补码再进行一次求补码的操作，才是其结果的原码。

计算机最终会选择补码进行存储和计算数字，这样做的优点如下:

  • 便于CPU统一使用加法运算器进行计算
  • 解决+0和-0的表达方式
  • 数字范围扩大成[-128, 127]

如果想让我的解决方式可以通过测试，那么还需要处理负数是的转换逻辑运算。这种方式处理起来比较麻烦，因此，如果感到实现方式比较繁琐，可能是我的思考角度不合适造成的。一个合适的思考维度，可以很大的降低代码的复杂度。

思维的迁移能力很重要!
